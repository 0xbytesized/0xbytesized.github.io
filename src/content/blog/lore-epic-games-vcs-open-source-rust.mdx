Hoy Epic Games ha hecho algo que no esperaba: abrir el código de su sistema de control de versiones interno. Se llama **Lore**, está escrito en Rust, tiene licencia MIT y llevaba años funcionando bajo el nombre de _Unreal Revision Control_ dentro del editor de Fortnite (UEFN). La noticia ha pegado fuerte en HN: 779 puntos y más de 430 comentarios en pocas horas.

No es un "Git killer". No pretende serlo. Lore apunta a un nicho muy concreto donde Git siempre ha hecho aguas: proyectos con miles de assets binarios grandes — texturas, modelos 3D, animaciones, audio. El territorio de Perforce.

## ¿Por qué otro VCS en 2026?

Si trabajas con código, Git te funciona. Pero en game dev la realidad es distinta. Un proyecto AAA puede tener terabytes de assets binarios. Una textura de 4K pesa 30 MB. Un modelo con animaciones, 200 MB. Git con LFS es un parche que funciona a medias: el diff de binarios no existe, el blame es inútil, y el rendimiento se degrada con repos grandes.

Perforce (Helix Core) ha sido el estándar de facto en la industria del videojuego durante décadas. Pero es propietario, caro, complejo de administrar y su UX es... digamos que no ha envejecido bien.

Epic Games llevaba años manteniendo su propio VCS interno para UEFN. Hoy lo han abierto.

## Qué hace Lore distinto

Lore tiene decisiones de diseño que lo separan de Git desde los cimientos:

### Binary-first, texto después

Git trata todo como texto y luego intenta manejar binarios con LFS. Lore invierte la premisa: todo es binario por defecto, y el texto es un caso especial que se maneja encima. Esto significa que un PNG de 30 MB y un `.cpp` de 2 KB reciben el mismo tratamiento en la capa de almacenamiento.

### Chunking inteligente con FastCDC

En lugar de almacenar snapshots completas de cada archivo, Lore trocea los binarios con FastCDC (_content-defined chunking_). Si modificas un píxel de una textura de 4K, solo se almacena el chunk que cambió, no la imagen entera. Esto es crucial cuando tienes cientos de revisiones de assets de 200 MB.

```rust
// Conceptual: cómo Lore divide un archivo binario en chunks
// usando content-defined chunking (FastCDC)
//
// No es código real de Lore, es una ilustración del concepto:
// FastCDC encuentra boundaries basados en el contenido,
// no en offsets fijos. Así, si insertas datos en medio
// del archivo, solo cambian los chunks afectados.

fn chunk_file(data: &[u8], min_size: usize, avg_size: usize, max_size: usize) -> Vec<Chunk> {
    let mut chunks = Vec::new();
    let mut start = 0;
    let mut i = min_size;

    while i < data.len() {
        // FastCDC usa un rolling hash para encontrar
        // boundaries naturales en el contenido
        if i >= max_size || (i >= min_size && is_cut_point(&data[start..i])) {
            chunks.push(Chunk {
                offset: start,
                length: i - start,
                hash: blake3(&data[start..i]),
            });
            start = i;
        }
        i += 1;
    }

    // Último chunk
    if start < data.len() {
        chunks.push(Chunk {
            offset: start,
            length: data.len() - start,
            hash: blake3(&data[start..]),
        });
    }

    chunks
}
```

### Content addressing con BLAKE3

Cada fragmento se identifica por su hash BLAKE3, no por su nombre de archivo. Esto permite deduplicación automática: si dos assets comparten datos idénticos (algo frecuentísimo en juegos), Lore solo los almacena una vez.

### API-first, no CLI-first

Git fue diseñado como un conjunto de comandos de terminal. Su "API" es llamar a `git` con flags. No existe una biblioteca linkable oficial — y es una decisión deliberada de sus mantenedores.

Lore hace lo contrario: la API es el artefacto principal. Existe `liblore`, una biblioteca compartida con bindings para Rust y JavaScript. El CLI es un cliente más que consume esa API. Esto permite integrar Lore en editores, pipelines de CI/CD y herramientas custom sin hacer parsing de stdout.

```bash
# Lore se instala con un script que baja los binarios precompilados
# CLI + servidor + biblioteca compartida

# macOS / Linux
curl -fsSL https://raw.githubusercontent.com/EpicGames/lore/main/scripts/install.sh | bash -s -- --demo

# Windows (PowerShell)
$env:LORE_DEMO = 1
irm https://raw.githubusercontent.com/EpicGames/lore/main/scripts/install.ps1 | iex
```

### Centralizado con capacidad offline

A diferencia de Git (distribuido) y más parecido a Perforce, Lore es centralizado por diseño. Pero no es un centralizado tonto: puedes trabajar offline, hacer commits locales y pushear cuando recuperes conexión. El servidor valida todo en el push, y el almacenamiento inmutable hace que la integridad de los datos sea verificable sin confianza ciega.

### Sparse por defecto

En un proyecto de 500 GB, ¿cuánto necesitas realmente en tu máquina? Lore hace _sparse checkout_ por defecto: solo descarga lo que necesitas. Los archivos se obtienen bajo demanda (_lazy fetch_) y se cachean localmente.

### Locks para assets no mergeables

Dos personas no pueden hacer merge de un archivo de Photoshop o de un modelo 3D como si fuera código. Lore incluye un sistema de locks para contenido binario no mergeable, algo que en Git simplemente no existe.

## El flujo de trabajo real

El quickstart de Lore te monta un entorno completo en 10 minutos. Así se ve el ciclo básico:

```bash
# 1. Instalar (ya trae servidor local)
curl -fsSL https://raw.githubusercontent.com/EpicGames/lore/main/scripts/install.sh | bash -s -- --demo

# 2. Crear repo
lore init mi-proyecto

# 3. Añadir archivos — un solo comando para adds, edits y deletes
echo "Hello, Lore" > hello.txt
lore stage hello.txt        # añade
lore stage move old.txt new.txt  # renombra
lore stage remove bye.txt   # elimina

# 4. Commit
lore commit -m "Primer revision"

# 5. Push al servidor
lore push

# 6. Clonar en otra working tree
lore clone lore://localhost:41337/mi-proyecto otro-directorio

# 7. Branch y merge
lore branch feature/x
lore switch feature/x
# ... cambios ...
lore commit -m "Feature X"
lore switch main
lore merge feature/x
```

La diferencia con Git más notable es `lore stage`: un solo comando para adds, edits, deletes, moves y renames. Nada de `git add`, `git rm`, `git mv` por separado. Lore detecta qué ha cambiado y actúa en consecuencia.

## Lo que NO es Lore

Para evitar malentendidos:

- **No es un reemplazo de Git para código.** Si tu proyecto son puros archivos de texto, Git te sigue sirviendo perfectamente.
- **No es completamente descentralizado.** Necesitas un servidor. Puede ser local, pero la arquitectura es centralizada.
- **No es un producto maduro.** Es v0.8.3. Acaba de salir. Los propios desarrolladores advierten que está en fase temprana.
- **El cliente de escritorio no es open source.** Solo los binarios del CLI, servidor y `liblore` lo son. El desktop client se distribuye como binario cerrado.

## Mi veredicto

Lore me parece una de esas movidas que tienen sentido estratégico aunque el producto aún esté verde. Epic Games no está haciendo caridad: abrir Lore les permite capturar contribuciones externas, crear ecosistema alrededor de su herramienta y, de paso, meter presión a Perforce en un mercado donde el pricing es un dolor constante para los estudios.

Técnicamente las decisiones son sólidas. BLAKE3 para content addressing, FastCDC para chunking, Zstd para compresión, Rust como lenguaje — no están improvisando. El diseño está documentado con un nivel de detalle que muchos proyectos open source consagrados ya quisieran: 17 secciones de system design, 16 ADRs públicos, especificaciones formales.

Lo que me frena de emocionarme demasiado: la dependencia del servidor. En un mundo donde Git te deja trabajar completamente offline y hacer push a cualquier remote, volver a un modelo centralizado sabe a paso atrás. Sí, Lore tiene capacidad offline, pero el modelo mental es "conectado con desconexiones toleradas", no "distribuido por naturaleza".

Dicho esto, para el caso de uso al que apunta — estudios de videojuegos, proyectos con assets binarios pesados, pipelines de producción donde necesitas locks y permisos por directorio — Lore resuelve problemas reales que Git simplemente no puede resolver sin una torre de herramientas auxiliares.

Le seguiré la pista. Si en un año llega a v1.0 con el desktop client abierto y un ecosistema de forjas como GitHub pero para Lore, la conversación será muy distinta.
