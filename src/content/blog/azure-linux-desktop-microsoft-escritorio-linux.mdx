Hayden Barnes, conocido en la comunidad WSL por su trabajo en Azure Linux y el ecosistema .NET, publicó esta semana un experimento que merece atención: **Azure Linux Desktop**, una aplicación Windows que arranca un escritorio Linux completo —XFCE, audio, aceleración GPU, portapapeles compartido— dentro de una ventana. Sin terminal, sin wizard de instalación, sin configuraciones. Abres la app y en unos segundos estás mirando un escritorio Linux.

No es un producto oficial de Microsoft. Es un hack de juguete, como él mismo admite. Pero lo interesante no es el resultado final: es **cómo une cuatro piezas presentadas en la Build 2026** que, por separado, ya apuntan a un cambio de fondo en la relación Windows-Linux.

## Las cuatro piezas del puzle

### 1. wslc: contenedores Linux nativos en Windows, con API y todo

`wslc` es el nuevo runtime de contenedores Linux para Windows. Corre contenedores OCI dentro de una VM Hyper-V dedicada, separada de tus distros WSL 2 habituales. Tiene CLI compatible con Docker, soporte para GPU vía CDI y —esto es lo relevante para el experimento— **un SDK completo en C/WinRT/C#** que permite a cualquier aplicación Windows crear, gestionar y comunicarse con contenedores Linux.

La API de wslc es lo que permite a Azure Linux Desktop arrancar la distro sin que el usuario toque una terminal. Los nombres de clases y métodos que siguen son ilustrativos (la API real usa `Microsoft.WSL.Containers` como namespace y métodos síncronos; consulta el [repositorio de WSL](https://github.com/microsoft/WSL) para la referencia exacta):

```csharp
// Ilustración conceptual del flujo de wslc.
// Los nombres de clases, namespaces y métodos son inventados
// para transmitir la idea. La API real usa Microsoft.WSL.Containers
// con métodos síncronos. Ver github.com/microsoft/WSL.

using Wslc.Sdk;  // namespace ilustrativo, el real es Microsoft.WSL.Containers

// 1. Crear una sesión wslc (ilustrativo; la API real es new Session() + .Start())
var session = await WslcSession.CreateAsync(new()
{
    VcpuCount = 2,
    MemoryMb = 2000
});

// 2. Cargar la imagen del escritorio
var imageId = await session.LoadImageAsync("azurelinux-desktop:latest");

// 3. Crear el contenedor (ilustrativo; el real es Session.CreateContainer síncrono)
var container = await session.CreateContainerAsync(new()
{
    ImageId = imageId,
    PortMappings = { (3389, 3389) },
    GpuAccess = GpuAccess.Enabled  // ilustrativo; el real es ContainerFlags.EnableGpu
});

// 4. Arrancar (ilustrativo; el real es Container.Start síncrono)
await container.StartAsync();
```

La comunicación con la VM va por Hyper-V sockets. Mensajes como `WSLC_FORK`, `WSLC_MOUNT` y `WSLC_UNIX_CONNECT` viajan multiplexados por un único canal. El montaje de carpetas Windows usa Plan 9 o virtiofs. La GPU se expone vía CDI v0.6.0. Todo esto ya está mergeado en el repo público de WSL, aunque no ha llegado a la versión estable.

### 2. WinUI Reactor: adiós al XAML

Una de las decisiones más llamativas del proyecto: **no hay un solo archivo .xaml en todo el repositorio**. Barnes usó WinUI Reactor, un framework nuevo que permite construir interfaces WinUI 3 con funciones componibles y hooks de estado en C#, al estilo React.

La aplicación entera —barra de título, pantalla de arranque, superficie del escritorio— se define en `App.cs` con factories de Reactor y cambios de estado:

```csharp
// Construcción declarativa de UI con WinUI Reactor
// Sin XAML: todo son funciones C# componibles

static VisualNode AppContent(AppState state)
{
    return state switch
    {
        AppState.Booting => new Grid()
        {
            new AzureLinuxLogo()
                .Size(120, 120),
            new BootSpinner()
                .IsActive(true)
        },

        AppState.Desktop => new Grid()
        {
            new DesktopSurface()
                .FillWindow(true)
        },

        _ => new TextBlock()
            .Text("Algo fue mal. Reinicia la app.")
            .FontSize(24)
    };
}
```

El `BootSpinner` merece mención aparte. No es un GIF. Es un conjunto de seis `SpriteShape` cuya posición se anima con `Vector3` keyframes en el compositor. Cada punto va desfasado en el tiempo para crear el efecto de órbita. Corre completamente en la GPU, sin ejecutar código managed por frame. Si te has preguntado alguna vez cómo hacer animaciones fluidas sin matar el rendimiento, aquí tienes un ejemplo de manual.

### 3. Azure Linux 4.0: la distro servidor aprende a llevar GUI

Azure Linux 4.0 es la distribución Linux de Microsoft, basada en un snapshot de Fedora 43. Está diseñada para servidores y contenedores: no trae paquetes de escritorio ni entorno gráfico. Pero como comparte linaje de paquetes con Fedora 43, Barnes pudo apuntar a los repositorios de Fedora e instalar XFCE desde ahí.

El resultado es un híbrido que "mayormente funciona", aunque requirió varios apaños. El más interesante tiene que ver con los iconos SVG:

```dockerfile
# Fragmento del Dockerfile del proyecto
# Fedora 43 dropeó el loader librsvg de gdk-pixbuf
# Ahora los SVG pasan por glycin, que usa bubblewrap
# y bubblewrap no puede crear user namespaces dentro del contenedor
# Solución: rasterizar todos los SVG a PNG durante la build

FROM azurelinux:4.0 AS builder
RUN dnf install -y xfdesktop librsvg2-tools

# Rasterizar iconos SVG del tema a PNG
RUN find /usr/share/icons -name "*.svg" -exec sh -c \
    'rsvg-convert -w 48 -h 48 "$1" -o "${1%.svg}.png"' _ {} \;

# Dejar bubblewrap instalado para que las llamadas fallidas
# no revienten nada — simplemente devuelven vacío
FROM azurelinux:4.0
COPY --from=builder /usr/share/icons /usr/share/icons
RUN dnf install -y xfce4-session xrdp bubblewrap
```

El audio por RDP también requirió ingeniería extra. Las sesiones de contenedor son VMs independientes, así que no hay acceso al audio de WSLg. La solución fue compilar `pipewire-module-xrdp` contra el PipeWire de la propia imagen y arrancarlo en secuencia después de `pipewire-pulse`.

### 4. .NET 10: publicar para x64 y ARM64 con un solo comando

La app targetea .NET 10 y WinUI 3, empaquetada con el Windows App SDK. `dotnet run` la lanza con identidad de paquete, ReadyToRun precompila para arranques más rápidos y un solo `dotnet publish` produce builds para x64 y ARM64.

Todo el proyecto cabe en un `.slnx` que se construye con `dotnet build`. Sin configuraciones complejas, sin pasos manuales. La madurez del tooling de .NET es probablemente la pieza menos llamativa del experimento, pero es la que permite que las otras tres encajen sin fricción.

## Los hacks que merecen aplauso

Más allá de la arquitectura limpia, el proyecto tiene soluciones ingeniosas donde las piezas no encajaban:

**El RDP no se lleva bien con WinUI.** El control ActiveX de Escritorio Remoto de Windows (`mstscax.dll`) se niega a pintar dentro de un `DesktopChildSiteBridge` y crashea si lo creas oculto o lo redibujas como `layered`. La solución es gloriosamente sucia: el control RDP vive en una ventana WinForms sin bordes, aparcada fuera de pantalla hasta que está lista y un timer de tracking la pega sobre la ventana principal sincronizando posición y tamaño en cada frame. No es elegante, pero funciona.

**La zona horaria del contenedor** se sincroniza convirtiendo el ID de zona horaria de Windows a su equivalente IANA y reescribiendo `/etc/localtime` al arrancar. El reloj del contenedor ya va por Hyper-V time synchronization, así que este paso solo corrige la etiqueta, no la hora en sí.

**El toque nostálgico.** El escritorio XFCE lleva el tema Bluecurve, el look que Red Hat estrenó en 2002. También incluye Visual Studio Code y PowerShell. Es Azure Linux, después de todo.

## Mi opinión

Lo que hace especial a este proyecto no es que Microsoft tenga un escritorio Linux —spoiler: no es un producto, es un hack de fin de semana—, sino lo que revela sobre la dirección técnica de la compañía.

En 2026, Microsoft tiene:
- Un runtime de contenedores Linux **nativo en Windows** con SDK público (wslc)
- Un framework de UI declarativo en C# que elimina XAML por completo (WinUI Reactor)
- Su propia distribución Linux basada en Fedora, usable como SO de propósito general (Azure Linux 4.0)
- Un stack de desarrollo (.NET 10 + Windows App SDK) que compila para x64 y ARM64 con un comando

Cada pieza es potente por separado. Pero juntas permiten que un desarrollador individual —no un equipo de Microsoft— construya en un fin de semana algo que hace cinco años habría requerido un equipo de ingeniería de sistemas y meses de trabajo.

Esa es la historia real aquí. La democratización de la infraestructura. El hecho de que puedas coger cuatro piezas presentadas en una conferencia, combinarlas en un proyecto personal y terminar con un escritorio Linux arrancando dentro de Windows como si fuera una app cualquiera. Con audio. Con GPU. Con portapapeles.

El escritorio Linux de Microsoft no es el producto. Es el síntoma de que la línea entre Windows y Linux se ha vuelto irrelevante para quien programa.

Y eso, francamente, mola.
