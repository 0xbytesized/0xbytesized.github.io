Hace unas horas Alistair Davidson publicó un artículo que ha reventado Hacker News con 738 puntos y más de 340 comentarios. Trabajaba como contractor para una compañía de servicios públicos — un monopolio regulado donde bajar del 96% de satisfacción son millones en multas. El problema: los clientes no conseguían completar el formulario de alta.

Habían intentado dos cosas antes de que llegara Alistair. Un formulario ASP clásico que envejecía mal y un proceso manual carísimo. Luego externalizaron una app en React a contratistas en otro país. Duró tres días online antes de que la quitaran por quejas. Alistair la revisó y le dijo a su jefe: "no podemos hacernos cargo de esto". Spinners por todas partes, estados globales en JavaScript, imágenes metidas en localStorage con un límite de 5 MB, cero accesibilidad.

Lo que hizo después es la parte que merece atención.

## La decisión: HTML funciona, lo demás es mejora progresiva

Construyó la nueva versión con [Astro](https://astro.build), un framework que genera HTML estático y te permite salpicar JavaScript solo donde hace falta. La idea era simple: el sitio debía funcionar sin JS. Punto. Si el navegador lo soportaba, se añadían mejoras cosméticas con web components, pero la experiencia base no dependía de nada más que HTML y un backend.

¿Su razonamiento? Una utility es un servicio público. Tus usuarios no están en un coworking con un MacBook Pro y fibra de 600 Mbps. Están en medio de una urbanización a medio construir con un Android de hace diez años comprado en el súper, tirando de 3G. Mandarles 20 MB de JavaScript antes de pintar un `<form>` es una salvajada.

El patrón es viejo pero efectivo: cada paso del wizard del formulario es su propia página. El usuario hace clic en "Siguiente", el form hace submit vía POST tradicional, el backend valida, guarda los datos en sesión y redirige a la página siguiente. Si todo falla, los datos ya están a salvo en el servidor. Nada de perder una solicitud a medio rellenar porque el teléfono se quedó sin batería.

## Código: validación nativa en menos de 1 KB

Alistair escribió un web component que envuelve cualquier `<form>` y reemplaza los popups de validación del navegador por mensajes inline accesibles. Lo llamó `validation-enhancer` y pesa menos de 1 KB. Si el navegador no soporta custom elements, el form sigue funcionando porque la validación nativa de HTML sigue ahí.

Aquí va una versión simplificada del concepto:

```html
<!-- Sin JS: el navegador valida con su UI nativa -->
<form method="post" action="/step-2">
  <label for="email">Correo electrónico</label>
  <input
    id="email"
    name="email"
    type="email"
    required
    aria-errormessage="email-error"
  />
  <div id="email-error" role="alert"></div>
  <button type="submit">Siguiente</button>
</form>
```

Y el custom element que lo mejora — un snippet de ~30 líneas que captura los eventos nativos del navegador y coloca los errores donde toca:

```javascript
class ValidationEnhancer extends HTMLElement {
  connectedCallback() {
    this.form = this.querySelector("form");
    if (!this.form) return;

    this.form.addEventListener("submit", (e) => {
      if (!this.form.checkValidity()) {
        e.preventDefault();
        this.showErrors();
      }
    });

    this.form.addEventListener("input", (e) => {
      const field = e.target;
      if (field.validity.valid) this.clearError(field);
    });
  }

  showErrors() {
    this.form.querySelectorAll(":invalid").forEach((field) => {
      const msgEl = document.getElementById(
        field.getAttribute("aria-errormessage")
      );
      if (msgEl) msgEl.textContent = field.validationMessage;
    });
  }

  clearError(field) {
    const msgEl = document.getElementById(
      field.getAttribute("aria-errormessage")
    );
    if (msgEl) msgEl.textContent = "";
  }
}

customElements.define("validation-enhancer", ValidationEnhancer);
```

Lo importante no es el código — es lo que representa. El equipo no pasó semanas peleándose con react-hook-form, Zod, Yup y cuatro capas de esquemas. Usaron lo que el navegador ya trae de serie y le pusieron un lazo de 1 KB encima.

## La comparación que nadie quiere hacer

Pongamos números encima de la mesa. Una SPA típica con React + form libraries:

| Recurso              | Tamaño estimado |
| -------------------- | --------------- |
| React + ReactDOM     | ~42 KB (gzip)   |
| react-hook-form      | ~9 KB (gzip)    |
| Zod + resolvers      | ~12 KB (gzip)   |
| Componentes, estado  | ~30-80 KB       |
| **Total**            | **90-140 KB**   |

La solución HTML-first:

| Recurso                     | Tamaño      |
| --------------------------- | ----------- |
| HTML del formulario         | ~2-5 KB     |
| CSS mínimo                  | ~3 KB       |
| Custom element validación   | ~1 KB       |
| **Total**                   | **~6-9 KB** |

No es un 20% más pequeño. Es un orden de magnitud. Y lo más bestia: si el JS falla — porque la red es mala, porque el navegador es viejo, porque un bloqueador de scripts se ha puesto agresivo — el formulario sigue funcionando. El backend recibe los datos igual. El usuario completa su trámite y sigue con su vida.

## Lo que pasó al desplegar

Cuando lanzaron la nueva versión, el número de formularios completados se duplicó. El equipo de analytics no entendía de dónde salían esos usuarios. La respuesta es incómoda: tu paquete de analytics basado en JavaScript no ve a los usuarios que estás rebotando por culpa del JavaScript que no se carga.

Hubo un caso que lo resume todo: alguien completó un formulario un mes después de empezarlo. Como cada paso guardaba los datos en sesión del lado del servidor, la persona pudo retomar exactamente donde lo dejó. Intenta hacer eso con un estado metido en un store de Zustand.

## Mi opinión

El artículo de Alistair ha pegado fuerte porque toca un nervio. Llevamos una década construyendo webs como si todo el mundo tuviera un iPhone 17 con 5G ilimitado. La realidad de la mayoría de usuarios — y hablo de España, de Europa, no de contextos extremos — es mucho más precaria de lo que el developer medio asume desde su oficina.

Lo que más me gusta del caso no es la tecnología. Es la claridad moral: "esto es un servicio público, debe funcionar para todo el mundo, sin excusas". Y luego ejecutó con herramientas que ya existen. No inventó un framework. No escribió un manifesto. Cogió Astro, HTML y un web component de 1 KB y duplicó las conversiones.

Hay una frase del artículo que me dejó pensando. Cuando Alistair le explicó a su reemplazo que el sitio funcionaba sin JavaScript, la respuesta fue "pero eso nos da más trabajo a nosotros". Ahí está el problema de fondo. No es técnico, es cultural. Nos hemos convencido de que la complejidad es inevitable y que los usuarios deben pagar el pato con cargas de 20 segundos y spinners infinitos.

No es verdad. HTML funciona. Lleva funcionando 30 años. Y el día que priorizas a los usuarios reales en lugar de a tu comodidad como desarrollador, los números hablan solos.
